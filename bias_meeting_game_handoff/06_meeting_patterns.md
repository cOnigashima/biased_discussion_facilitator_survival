# Meeting / Scenario Patterns

## Overview

ゲーム全体は、ソフトウェア開発組織におけるさまざまな会議・意思決定をステージ化できる。

各ステージは以下の型で作る。

```text
舞台
役職
対立軸
主要キャラ
主なバイアス
探索資料
バトルパターン
勝ち筋
失敗例
```

## Pattern A: リリース判定会議

| Item | Description |
|---|---|
| 舞台 | リリース前日のGo/No-Go会議 |
| プレイヤー | シニアエンジニア、Tech Lead、QA Lead |
| 敵っぽい人 | PM、営業、CTO、Tech Lead |
| 主なバイアス | 正常性、サンクコスト、計画錯誤、確証 |
| 探索資料 | Slack、Jira、CI、PR、ロールバック手順、顧客メール |
| カタルシス | 全面リリース/全面延期の二択を、分割リリースに変える |
| 失敗例 | 強行リリースして障害、または過剰延期で顧客信頼低下 |

よくある発言：

- 「前回も同じような警告は出ていました」
- 「ここまで来て止めるのは現実的じゃないです」
- 「これはflaky testです」
- 「顧客には今月中と伝えています」
- 「全部止めるべきです」

## Pattern B: 障害対応ウォールーム

| Item | Description |
|---|---|
| 舞台 | 本番障害中の緊急会議 |
| プレイヤー | SRE、オンコール、Incident Commander |
| 敵っぽい人 | 焦る役員、責任逃れするチーム、直近原因に固執するベテラン |
| 主なバイアス | 利用可能性、正常性、確証、責任分散 |
| 探索資料 | Datadog/Grafanaログ、deploy履歴、アラート、runbook、Slack |
| カタルシス | 混乱を役割分担と仮説検証に変える |
| 失敗例 | 直近原因に引っ張られて誤対応 |

ギミック：

- Timeが厳しい。
- 質問しすぎると復旧が遅れる。
- 即断しすぎると誤対応。
- Blame Momentumが発生しやすい。

## Pattern C: ポストモーテム裁判

| Item | Description |
|---|---|
| 舞台 | 障害後の振り返り会 |
| プレイヤー | ファシリテーター、EM、SRE Lead |
| 敵っぽい人 | 「だから言ったのに」系シニア、責任転嫁PM、怒る役員 |
| 主なバイアス | 後知恵、自己奉仕、基本的帰属の誤り、ホーン効果 |
| 探索資料 | 当時Slack、判断ログ、アラート履歴、PR、runbook |
| カタルシス | 犯人探しを構造改善に変える |
| 失敗例 | スケープゴート化、チーム信頼低下 |

よくある発言：

- 「あのログを見れば分かっていたはずです」
- 「QAがもっと早く止めるべきでした」
- 「この担当者の確認不足です」

## Pattern D: アーキテクチャレビュー

| Item | Description |
|---|---|
| 舞台 | マイクロサービス化、基盤刷新、AI導入、巨大リライト |
| プレイヤー | Tech Lead、Staff Engineer、Architect |
| 敵っぽい人 | 流行技術推しCTO、技術顧問、元有名企業エンジニア |
| 主なバイアス | 権威、バンドワゴン、生存者、アンカリング、確証 |
| 探索資料 | RFC、ADR、運用体制、障害履歴、採用状況、コスト試算 |
| カタルシス | 技術宗教戦争を制約とトレードオフに戻す |
| 失敗例 | 大企業の真似をして運用崩壊 |

よくある発言：

- 「有名企業でもこの構成です」
- 「今どきこの技術を使わないのは遅れています」
- 「この設計ならスケールします」

## Pattern E: ブリリアントジャーク対策

| Item | Description |
|---|---|
| 舞台 | 高成果エンジニアがレビューや会議で周囲を壊している |
| プレイヤー | EM、Tech Lead、評価者 |
| 敵っぽい人 | 圧倒的に実装できるが、周囲を萎縮させるエース |
| 主なバイアス | ハロー効果、成果バイアス、自己奉仕、基本的帰属の誤り |
| 探索資料 | PRレビュー履歴、退職兆候、1on1メモ、属人化マップ、評価シート |
| カタルシス | 「あの人がいないと無理」という空気を崩す |
| 失敗例 | 優秀な凡人が辞め、ジャークだけが残る |

よくある発言：

- 「厳しくしないと品質が落ちる」
- 「できない人に合わせているから遅い」
- 「俺が直した方が早い」

## Pattern F: 顧客・クライアント折衝

| Item | Description |
|---|---|
| 舞台 | 顧客が無理な納期・仕様変更を要求する |
| プレイヤー | ITコンサル、PM、プリセールス、Tech Lead |
| 敵っぽい人 | 顧客役員、情シス部長、自社営業、競合コンサル |
| 主なバイアス | アンカリング、損失回避、フレーミング、権威、サンクコスト |
| 探索資料 | 契約、提案書、顧客メール、議事録、現場ヒアリング |
| カタルシス | クレーム会議を実行可能な合意に変える |
| 失敗例 | 「できます」と言って地獄、または正論で顧客を怒らせる |

## Pattern G: 技術インフルエンサー汚染

| Item | Description |
|---|---|
| 舞台 | 社内がSNSやカンファレンスの流行に影響される |
| プレイヤー | Tech Lead、Staff Engineer、技術戦略担当 |
| 敵っぽい人 | インフルエンサー信者、流行好きCTO、採用広報 |
| 主なバイアス | バンドワゴン、権威、FOMO、生存者、利用可能性 |
| 探索資料 | ブログ、カンファレンス資料、自社運用体制、PoC結果 |
| カタルシス | キラキラ技術を使いどころへ落とす |
| 失敗例 | 流行技術導入で運用崩壊、または保守的すぎて機会損失 |

## Pattern H: 評価・昇進会議

| Item | Description |
|---|---|
| 舞台 | 昇進、降格、評価、配置転換 |
| プレイヤー | EM、部長、人事、Tech Lead |
| 敵っぽい人 | 声が大きい評価者、過去の印象に引っ張られる上司 |
| 主なバイアス | ハロー、ホーン、直近効果、基本的帰属の誤り |
| 探索資料 | 評価シート、PR履歴、1on1メモ、チーム影響、成果記録 |
| カタルシス | 雰囲気評価を構造化して公平な判断に近づける |
| 失敗例 | 目立つ人だけ昇進し、静かに支える人が腐る |

## Pattern I: ジュニア育成・1on1

| Item | Description |
|---|---|
| 舞台 | 自信過剰または自信喪失している若手との1on1 |
| プレイヤー | メンター、Tech Lead、EM |
| 敵っぽい人 | 「AIで全部作れます」系若手、または「自分は何もできない」系若手 |
| 主なバイアス | ダニング＝クルーガー、確証、ホーン、学習性無力感 |
| 探索資料 | PR、レビューコメント、本人メモ、1on1履歴 |
| カタルシス | 偏った自己認識を現実的な成長計画に変える |
| 失敗例 | 潰す、甘やかす、放置する |

## Pattern J: 経営会議・事業撤退判断

| Item | Description |
|---|---|
| 舞台 | 赤字プロダクト、巨大リライト、撤退、ピボット、採用凍結 |
| プレイヤー | CTO、VPoE、事業責任者、技術顧問、社外コンサル |
| 敵っぽい人 | CEO、CFO、投資家、事業部長、古参メンバー |
| 主なバイアス | サンクコスト、確証、自己奉仕、正常性、楽観 |
| 探索資料 | P/L、顧客データ、技術負債、採用計画、ロードマップ |
| カタルシス | 続ける/やめるの感情戦を、条件付き判断に変える |
| 失敗例 | 死に筋事業に人を貼り続ける、または早すぎる撤退 |

## Difficulty Progression

| Level | Situation | Design Notes |
|---|---|---|
| Lv1 | 相手が露骨に偏っている | チュートリアル。発言と証拠の矛盾が明確 |
| Lv2 | 相手の言い分に一理ある | 証拠だけでなく出し方が必要 |
| Lv3 | 複数人が別方向に偏る | MomentumとSupportを扱う |
| Lv4 | 善意の人が間違える | 共感・退路が重要 |
| Lv5 | プレイヤー側にも偏りがある | 技術リスク偏重などが結果に出る |
| Lv6 | どの選択にも損失がある | Safety/Delivery/Trustのトレードオフ |
| Lv7 | 相手が賢く反論してくる | 罠証拠や反撃が増える |
| Lv8 | 政治・金・評価・顧客が絡む | 経営会議、撤退判断、組織再編 |
