# Codex / Next Session Handoff Prompt

以下の企画を引き継ぎ、プロトタイプ設計または実装に進める。

## 企画概要

ソフトウェア開発プロジェクトを舞台にした、探索パート + 会議バトル型ADV / 推理ゲーム。

プレイヤーはSlack、Jira、PR、CIログ、障害ログ、顧客メール、会議メモ、過去議事録などを読み、証拠カードや過去発言カードを集める。バトルパートでは、会議中の発言や議事録に対して、証拠・質問・共感・フレーム変換・退路提示などを使う。目的は相手を単に論破することではなく、会議で誤った前提が固まる前に止め、より現実的な意思決定案を通すこと。

## 参照感

- 逆転裁判：発言の怪しい箇所を選び、証拠を突きつける。
- ダンガンロンパ：会話中の発言をタイミングよく撃つ。
- Detroit: Become Human：状態値・フラグ・選択で流れや結末が変わる。
- Papers, Please：書類から違和感を見つける。

完全分岐木は制作量が爆発するため、基本は「固定ノード + 状態値で台詞・選択肢・結末が変わる」方式。

## 最有力MVP

第1ステージだけ作る。

タイトル仮：

- `Go / No-Go: 決済v2 リリース判定会議`
- `Release Judgment: 決済移行前夜`

プレイヤーはシニアエンジニア。明日、新しい決済フローをリリースする予定だが、CI失敗、PRコメント、staging障害ログ、顧客メール、ロールバック手順に不穏なズレがある。

登場人物：

- PM 佐伯：リリースしたい。サンクコスト、正常性、計画錯誤。
- Tech Lead 黒瀬：設計を守りたい。確証、自己奉仕、権威。
- QA 三村：全面停止したい。損失回避、利用可能性。
- 営業 桐谷：顧客約束を守りたい。アンカリング、フレーミング、損失回避。
- CTO 榊：成果を出したい。楽観、権威、正常性。

会議ラウンド：

1. PMが「前回も似た警告だった」と主張する。
2. Tech Leadが「これはflaky test」と主張する。
3. 営業が「顧客には今月中と伝えている」と主張する。
4. QAが「全部止めるべき」と主張する。
5. CTOが最終判断を求める。
6. 可能なら議事録修正フェーズを入れる。

探索カード：

- CI失敗ログ：`legacy_plan=true` かつ `tax_category=null` で再現。
- PRコメント：legacy customer の `tax_category=null` への懸念。
- staging障害ログ：同条件で500。
- 顧客メール：今月中に必要なのは請求書出力で、新決済フロー全体ではない。
- ロールバック手順：customer_data_migration は完全ロールバック不可。
- Feature Flag設計：checkout_v2 UI と invoice export は制御可能。
- 過去障害報告：似た兆候の軽視で障害が起きたことがある。
- CPUスパイクログ：罠。今回の核心とは無関係。

最良エンド：

- invoice exportのみ段階リリース。
- checkout_v2 UIはfeature flag off。
- customer_data_migrationは延期。
- 次回以降、ロールバック不能な変更をGo/No-Go前に明示する。
- 同一条件で3回以上落ちるCIはflaky扱いしないルールを作る。

## バトルの重要概念

- 発言ベース：現在の発言の怪しい部分を選び、証拠や質問を使う。
- 議事録ベース：発言が議事録に積まれ、放置すると会議の前提になる。
- Premise Lock：怪しい発言が、発言 → 未検証論点 → 暫定前提 → 合意済み前提 → 決定根拠へ固まる。
- Momentum：会議の流れ。Release寄り、Stop寄り、Split Release寄り、Blame Game寄りなど。
- ピン留め：今は突かずに議事録に残し、後で別証拠や別発言と組み合わせて刺す。
- 議事録修正：最後に自動議事録の誤った要約を修正し、最終判断を変える。

## 重要パラメータ

会議全体：

- Heat：荒れ具合。高いと証拠が通りにくい。
- Clarity：論点の明確さ。高いと良い選択肢が解放される。
- Support：プレイヤー案への支持。高いと最終案が通る。
- Time：残り発言機会。
- Momentum：会議の流れ。
- Premise Lock：怪しい前提の固定度。

キャラごと：

- 確信度
- 防衛心
- 面子
- 不安
- 権威
- プレイヤーへの信頼
- 利害

## プレイヤーコマンド

- 証拠提示
- 質問
- 共感
- フレーム変換
- 退路提示
- 議事録にピン留め
- 議事録修正
- 味方に振る

## 注意点

- バイアス名当てクイズにしない。
- ラベル貼りはゲーム内で悪手にしやすい。
- 抽象的なテーマ語りに逃げない。プレイヤーが何を見て、何を選び、何が変わるかを具体化する。
- キャラは馬鹿にしない。どのキャラにも一理ある主張、利害、恐れ、守りたいものを持たせる。
- まだ「何が一番面白いか」は仮説段階。MVPでは「探索資料から違和感を見つけ、会議でタイミングよく使う」遊びが成立するかを検証する。
