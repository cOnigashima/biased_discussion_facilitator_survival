# Glossary

## Game Terms

### 探索パート

Slack、Jira、PR、ログ、顧客メールなどを読み、証拠カードや人物情報を集めるパート。

### バトルパート

会議中の発言や議事録に対して、証拠・質問・共感・フレーム変換・退路提示などを使うパート。

### 発言ベース

現在の発言の怪しい箇所を選び、証拠やアクションを使う仕組み。

### 議事録ベース

発言が議事録に蓄積され、放置すると前提化する仕組み。

### 証拠カード

探索パートで得た資料。会議で使う武器。

### 過去発言カード

会議中に誰かが言った発言をピン留めしたもの。後で証拠と組み合わせて使う。

### ピン留め

発言をその場では突かず、議事録に保存して後で使うこと。

### Premise Lock

怪しい発言が、未検証論点から暫定前提、合意済み前提、決定根拠へ固まっていく状態。

### Momentum

会議の流れ。Release寄り、Stop寄り、Split Release寄り、Blame Game寄りなど。

### Heat

会議の荒れ具合。

### Clarity

論点の明確さ。

### Support

プレイヤー案への支持。

### Time

残り発言機会。

### 退路提示

相手の面子を守りながら、主張を修正できる道を作る行動。

### フレーム変換

「出す/止める」などの二択や言葉の誘導を崩し、別の切り方に変える行動。

## Software Terms

### Go/No-Go

リリースするかしないかを判断する会議や判断行為。

### flaky test

実装が壊れていなくても不安定に失敗するテスト。ただし、条件が揃って再現する場合はflakyではなく実バグの可能性がある。

### Feature Flag

機能を設定で有効/無効にする仕組み。分割リリースやロールバックに使える。

### Rollback

リリース後に問題が起きたとき、前の状態へ戻すこと。

### migration

データ構造やデータ内容を新仕様へ変換する処理。不可逆なwriteが含まれるとリスクが高い。

### PR / Pull Request

コード変更をレビューする単位。過去の懸念コメントが証拠になる。

### staging

本番に近い検証環境。

### ADR

Architecture Decision Record。設計判断の記録。

### RFC

Request for Comments。設計や方針の提案文書。

## Bias Terms

### 正常性バイアス

異常を「いつも通り」と処理する傾向。

### 確証バイアス

自説に合う情報を重視し、反証を軽視する傾向。

### サンクコスト効果

すでに投じたコストを惜しんで続ける傾向。

### 損失回避

同じ大きさの利益より損失を強く嫌う傾向。

### アンカリング

最初に出た数字や条件に引っ張られる傾向。

### フレーミング効果

表現のされ方で判断が変わる傾向。

### 後知恵バイアス

結果を知った後で「最初から分かっていた」と感じる傾向。

### ハロー効果

一つの良い特徴で全体評価が上がる傾向。

### ホーン効果

一つの悪い特徴で全体評価が下がる傾向。

### 基本的帰属の誤り

他人の失敗を状況ではなく性格や能力のせいにしすぎる傾向。

### 生存者バイアス

成功例だけを見て、失敗例を無視する傾向。

### 権威バイアス

肩書きや有名人の発言を過信する傾向。

### バンドワゴン効果

多数派や流行に流される傾向。

### 計画錯誤

作業時間やコストを甘く見積もる傾向。

### バイアス盲点

他人のバイアスには気づくが、自分のバイアスには気づきにくい傾向。
