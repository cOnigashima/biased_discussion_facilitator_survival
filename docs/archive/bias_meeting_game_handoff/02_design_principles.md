# Design Principles

## 1. 抽象化より、プレイヤー行動を先に書く

避けたい方向：

- 「このゲームの本質は……」と抽象的にまとめる。
- 「人間は〜」と一般論に逃げる。
- バイアスの道徳的解説になる。

優先する方向：

- プレイヤーは何を見るのか。
- どの発言を選ぶのか。
- どのカードを使うのか。
- その結果、会議のどの状態が変わるのか。
- 何がエンディングに影響するのか。

## 2. バイアス名は前面に出さない

プレイ中にやることは、バイアス名を当てることではない。

悪い例：

```text
PMに正常性バイアスが発生！
サンクコスト効果を解除せよ！
```

良い例：

```text
PM「前回も似た警告は出ていましたけど、問題ありませんでしたよね」
```

プレイヤーは、前回ログと今回CIログを比較して、「同じではない」と突く。

バイアス名は以下に回す。

- クリア後リザルト
- 図鑑
- 追加解説
- プレイヤーが深掘りしたいときの補足

## 3. 論破より、会議の状態変更を重視する

「相手を黙らせる」は一部の勝ち方でしかない。

より重要なのは：

- 誤った前提を未確定に戻す。
- リスクを可視化する。
- 顧客要求を分解する。
- 面子を潰さずに方向転換させる。
- 支持者を増やす。
- 最終議事録を書き換える。
- 次回のルールを作る。

## 4. 正しい証拠も、出し方を間違えると失敗する

同じCIログでも、出し方で結果が変わる。

```text
悪手：
「黒瀬さんの設計ミスですよね？」

良手：
「設計全体を否定したいわけではなく、legacy顧客だけ条件漏れしていないか見たいです」
```

証拠の強さだけでなく、以下を扱う。

- タイミング
- 相手の防衛心
- 面子
- 会議Heat
- Support
- Momentum
- 議事録での前提化

## 5. キャラには必ず「一理ある主張」を持たせる

単なる愚か者にしない。

例：

- PMは評価を守りたいだけでなく、顧客とチームの努力も守りたい。
- Tech Leadは自己防衛しているが、本当に設計経験がある。
- QAは過剰停止に見えるが、過去に責められた経験がある。
- 営業は邪魔者ではなく、顧客信用を守ろうとしている。
- CTOは短期成果に偏るが、事業継続の責任を負っている。

## 6. ソフトウェア開発の具体性を武器にする

抽象論より、具体的な現場アイテムを使う。

- CI failure
- flaky test
- staging log
- migration
- rollback
- feature flag
- PR review
- Jira priority
- customer scope
- Go/No-Go criteria
- incident postmortem
- review severity
- ADR / RFC
- SLO / error budget

## 7. 「会議の前提が固まる怖さ」を遊びにする

誰かの発言が放置されると、会議の前提になる。

```text
発言
→ 未検証論点
→ 暫定前提
→ 合意済み前提
→ 決定根拠
```

プレイヤーは、どの発言をすぐ止めるべきか、どれをピン留めして後で使うべきかを判断する。

## 8. 教育効果は副産物として扱う

学びはあってよいが、表面上は「面白い会議バトル」であることを優先する。

望ましい順序：

1. 資料を読んで違和感を見つける。
2. 会議で刺す。
3. 分岐が変わる。
4. 結果として「これは正常性バイアスだったのか」と後で分かる。

## 9. Detroit風分岐は「見た目」と「重要局面」に限定する

完全分岐木は制作量が大きすぎる。

実装方針：

- 大きなラウンド構造は固定。
- 状態値で台詞、選択肢、反撃、エンドを変える。
- 重要局面だけ明確に分岐。
- 最後にフローチャートで「別ルートがあった」感を出す。

## 10. 最初の体験は小さく鋭く

MVPは1ステージでよい。

必要なのは、10〜20分で以下が体験できること：

- 探索で違和感を見つける。
- 会議で発言を突く。
- すぐ撃つか、ピン留めするか迷う。
- 議事録が誤った前提を固めようとする。
- 最後に分割リリース案を通す。
- リザルトで「あの発言を放置したらBadだった」と分かる。
