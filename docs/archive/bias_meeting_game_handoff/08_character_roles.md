# Character Roles and Archetypes

## Character Creation Formula

```text
役職
× 守りたいもの
× 恐れているもの
× 表の正論
× バイアス3つ
× 引けない理由
× 退路
```

例：

```text
役職: Tech Lead
守りたいもの: 自分の設計、チームの速度
恐れているもの: 技術判断ミスの露呈
表の正論: この設計が一番スケールする
バイアス: 確証、権威、サンクコスト
引けない理由: CTOにすでに説明済み
退路: 設計撤回ではなく検証フェーズ追加
```

## Stage 1 Characters

### PM: 佐伯

| Field | Value |
|---|---|
| Role | Project Manager |
| Stance | リリースしたい |
| Surface Claim | ここまで来たので予定通り出したい |
| Hidden Fear | 延期で評価が落ちる、顧客説明で詰む |
| Protects | 納期、チームの努力、自分の評価 |
| Bias Stack | サンクコスト、正常性、計画錯誤、自己奉仕 |
| Trigger | 「無責任」「見通しが甘い」と言われる |
| Good Approach | 前回との差分、未来コスト、分割案 |
| Escape Route | 全面延期ではなくフェーズ切り直し |

### Tech Lead: 黒瀬

| Field | Value |
|---|---|
| Role | Tech Lead |
| Stance | 自分の設計を守りたい |
| Surface Claim | これはflaky testで実装問題ではない |
| Hidden Fear | 設計ミスを認めると権威が下がる |
| Protects | 技術的評価、設計の正当性、チーム速度 |
| Bias Stack | 確証、自己奉仕、権威、ハロー |
| Trigger | 「設計ミスですよね」と言われる |
| Good Approach | 設計全体ではなくlegacy条件だけ確認 |
| Escape Route | 追加検証・限定修正・条件漏れ確認 |

### QA: 三村

| Field | Value |
|---|---|
| Role | QA Lead |
| Stance | リリースを止めたい |
| Surface Claim | 危険なので全部止めるべき |
| Hidden Fear | 前回障害でQAが責められた記憶 |
| Protects | 品質、顧客、QAチームの信用 |
| Bias Stack | 損失回避、利用可能性、ホーン寄り |
| Trigger | 「過剰反応」と言われる |
| Good Approach | 懸念を認め、止める対象を分ける |
| Escape Route | migrationだけ止める、レビュー条件を明確化 |

### Sales: 桐谷

| Field | Value |
|---|---|
| Role | Sales / Account Manager |
| Stance | 顧客約束を守りたい |
| Surface Claim | 顧客には今月中と伝えている |
| Hidden Fear | 顧客信用低下、売上未達、営業責任 |
| Protects | 顧客関係、契約、売上 |
| Bias Stack | アンカリング、フレーミング、損失回避 |
| Trigger | 「営業が勝手に約束した」と言われる |
| Good Approach | 顧客が必要な範囲を分解する |
| Escape Route | 請求書出力だけ出せば説明できる |

### CTO: 榊

| Field | Value |
|---|---|
| Role | CTO |
| Stance | 成果は出したいが事故は避けたい |
| Surface Claim | 完璧を求めすぎると何も出せない |
| Hidden Fear | 事業スピード低下、経営陣からの圧力 |
| Protects | 事業速度、技術組織の信用 |
| Bias Stack | 楽観、権威、正常性、フレーミング |
| Trigger | 「技術リスクだけ」を延々話される |
| Good Approach | 安全に出す案、顧客約束も守る案 |
| Escape Route | 分割リリース + Go/No-Go基準化 |

## Broader Archetypes

### リリース強行PM

```text
表の主張: ここまで来たんだから出しましょう
裏の恐れ: 延期で自分の評価が下がる
バイアス: サンクコスト、正常性、計画錯誤、自己奉仕
悪手: あなたは無責任です
良手: 今後失う可能性があるものを先に見積もりませんか
退路: 全面延期ではなくフェーズ分割
```

### 権威系テックリード

```text
表の主張: この設計が業界標準です
裏の恐れ: 技術判断が古いと思われる
バイアス: 権威、確証、バンドワゴン、アンカリング
悪手: それ流行に乗ってるだけですよね
良手: その企業と自社で違う条件を洗い出しましょう
退路: 全面採用ではなく一部領域で検証
```

### ブリリアントジャーク

```text
表の主張: 俺が厳しいことを言わないと品質が落ちる
裏の恐れ: 自分の価値が下がる、凡庸なチームに飲まれる
バイアス: ハロー、成果バイアス、自己奉仕、基本的帰属の誤り
悪手: あなたは性格が悪い
良手: 品質基準と人格攻撃を分けましょう
退路: レビュー基準を文書化し、暗黙知をチーム資産にする
```

### レガシー守護神

```text
表の主張: このシステムは10年これで動いてきた
裏の恐れ: 変更による障害、過去の自分の否定
バイアス: 現状維持、損失回避、正常性、利用可能性
悪手: 古いですね
良手: 壊さないために、どこからなら変えられますか
退路: 段階移行、並行稼働、ロールバック計画
```

### 技術顧問

```text
表の主張: 私の経験上、この会社には大規模刷新が必要です
裏の恐れ: 顧問としての価値を示したい
バイアス: 権威、フレーミング、アンカリング、生存者
悪手: 外部の人には分からない
良手: 成功条件と失敗条件を分けて比較しましょう
退路: 刷新ではなく診断フェーズを設ける
```

### ITコンサル

```text
表の主張: このロードマップなら半年でDXできます
裏の恐れ: 受注したい、役員に刺さる絵を出したい
バイアス: フレーミング、権威、アンカリング、計画錯誤
悪手: コンサル資料っぽいだけですよね
良手: 現場の制約と移行順序を具体化しましょう
退路: PoC、限定導入、成果指標の再定義
```

### テックインフルエンサー信者

```text
表の主張: 今どきこれを使わないのは遅れています
裏の恐れ: 自分が時代遅れになること
バイアス: バンドワゴン、権威、FOMO、確証
悪手: SNSに影響されすぎ
良手: その技術が刺さる条件を整理しよう
退路: 本番導入ではなく技術検証枠
```

### 調和型マネージャー

```text
表の主張: みんなで話し合って決めよう
裏の恐れ: 対立、退職、責任を負うこと
バイアス: 同調、責任分散、正常性、現状維持
悪手: 決断できないだけですよね
良手: 決めないことで誰が困るかを見ましょう
退路: 判断基準を先に決め、決定責任を明確にする
```

### データ絶対主義者

```text
表の主張: 数字で出ているので正しいです
裏の恐れ: 曖昧な判断をしたくない
バイアス: フレーミング、測定バイアス、確証、アンカリング
悪手: 数字だけ見ても意味ない
良手: その数字が拾っていないものは何ですか
退路: 補助指標と定性情報を追加
```

## Boss Shields

| Shield | Description | Counter |
|---|---|---|
| 権威シールド | 肩書きや有名企業実績で守る | 前提条件比較 |
| 面子シールド | 人前で撤回できない | 退路提示、1on1 |
| 善意シールド | 「チームのため」「顧客のため」 | 善意を認め、副作用を見る |
| 成果シールド | 過去成果で現在の害が見えない | 成果と副作用を分離 |
| 専門用語煙幕 | 難しい言葉で押し切る | 平易な説明、具体化 |
| 顧客圧シールド | 「顧客が言っている」 | 目的・手段・期限に分解 |
| 数字シールド | 都合のよいKPIだけ見せる | 分母・期間・除外条件を見る |
| 緊急性シールド | 「時間がない」で打ち切る | 今決めること/保留を分ける |
