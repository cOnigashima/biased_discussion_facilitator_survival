# Bias Catalog for Gameplay

## Usage Policy

バイアスはゲーム中に名前を直接出すより、発言・行動・議事録・キャラの利害として表現する。名前はリザルトや図鑑に回す。

## Core Bias Table

| Bias | 一言 | Game Expression | Common Counter |
|---|---|---|---|
| 正常性バイアス | 異常をいつも通り扱う | 「前回も大丈夫だった」 | 前回との差分、発生条件 |
| 確証バイアス | 自説に合う情報だけ見る | 「ほら、やっぱり」 | 反証、逆仮説、条件確認 |
| サンクコスト効果 | ここまでやったから続ける | 「今やめたら全部無駄」 | 今後コスト、ゼロベース質問 |
| 現状維持バイアス | 変えない方を過大評価 | 「今のままで困ってない」 | 変えないコスト、段階移行 |
| 損失回避 | 損を強く嫌う | 「失敗したらどうする」 | 損益の対称比較 |
| アンカリング | 最初の数字・条件に縛られる | 「今月中と決めた」 | 基準の妥当性確認 |
| 利用可能性ヒューリスティック | 思い出しやすい例を過大評価 | 「前も同じ事故があった」 | 発生率、差分確認 |
| 後知恵バイアス | 結果後に分かっていた気になる | 「あの時点で当然分かった」 | 当時情報、意思決定ログ |
| 生存者バイアス | 成功例だけ見る | 「有名企業もやっている」 | 失敗例、前提条件比較 |
| 権威バイアス | 肩書きや実績を過信 | 「顧問が言っている」 | 主張と人物の分離 |
| バンドワゴン効果 | 流行・多数派に流される | 「界隈では常識」 | 自社条件、独立評価 |
| ハロー効果 | 一つの良さで全体高評価 | 「あの人は優秀だから」 | 評価軸の分解 |
| ホーン効果 | 一つの悪さで全体低評価 | 「あのミスをした人だ」 | 事実と印象の分離 |
| 自己奉仕バイアス | 成功は自分、失敗は環境 | 「自分は悪くない」 | 責任分解、寄与度整理 |
| 基本的帰属の誤り | 他人の失敗を性格のせいにする | 「やる気がない」 | 状況要因、制約確認 |
| 計画錯誤 | 時間やコストを甘く見る | 「一週間でいける」 | 過去実績、類似案件 |
| フレーミング効果 | 表現で判断が変わる | 「延期 = 信用失墜」 | 言い換え、目的/手段分離 |
| ダニング＝クルーガー効果 | 知らないから自信過剰 | 「簡単でしょ」 | 知らないことの可視化 |
| バイアス盲点 | 自分の偏りに気づかない | 「自分は客観的」 | 第三者視点、記録 |

## Bias Stacking Patterns

### リリース強行PM

| Bias Stack | Effect |
|---|---|
| サンクコスト + 正常性 + 計画錯誤 + 自己奉仕 | 出したがる。リスクを軽く見て、延期による評価低下を恐れる。 |

発言：

```text
ここまで来て止めるのは現実的じゃないです。
前回も似た警告はありましたけど、問題ありませんでした。
```

攻略：

- 前回との差分を比較する。
- 今後失うものを見積もる。
- 全面延期ではなく分割案を出す。

### 権威系Tech Lead

| Bias Stack | Effect |
|---|---|
| 確証 + 自己奉仕 + 権威 + ハロー | 自分の設計を守る。懸念をテスト側の問題にする。 |

発言：

```text
これはflaky testです。この設計は前職でも使っていました。
```

攻略：

- 設計全体ではなく特定条件に絞る。
- 失敗条件の再現性を見る。
- 面子を守る言い方を使う。

### 損失回避QA

| Bias Stack | Effect |
|---|---|
| 損失回避 + 利用可能性 + 正常性反転 | 過去障害の記憶が強く、全面停止に寄りがち。 |

発言：

```text
前回もこういう軽視から障害になりました。全部止めるべきです。
```

攻略：

- 懸念を認める。
- 危険部分と安全部分を分ける。
- ロールバック不能な変更を特定する。

### 顧客圧営業

| Bias Stack | Effect |
|---|---|
| アンカリング + フレーミング + 損失回避 | 「今月中」という約束に引っ張られる。 |

発言：

```text
顧客には今月中と伝えています。延期したら信用を失います。
```

攻略：

- 顧客が本当に必要としている範囲を確認。
- 顧客約束を守る別手段を出す。
- 営業を敵ではなく味方に変える。

### ブリリアントジャーク

| Bias Stack | Effect |
|---|---|
| ハロー + 成果バイアス + 自己奉仕 + 基本的帰属の誤り | 成果があるため、周囲への害が見えにくい。 |

発言：

```text
厳しくしないと品質が落ちます。できない人に合わせる意味はありますか？
```

攻略：

- 品質基準と人格攻撃を分ける。
- レビュー指摘の重大度を分類。
- 暗黙知を文書化。
- 属人化のリスクを可視化。

### 技術顧問 / インフルエンサー型

| Bias Stack | Effect |
|---|---|
| 権威 + 生存者 + バンドワゴン + アンカリング | 他社成功事例や肩書きで大型提案を押す。 |

発言：

```text
他社ではこの方式で成功しています。今やらないと競争力を失います。
```

攻略：

- 成功条件と失敗条件を分ける。
- 自社の運用体制、採用力、トラフィック、リリース頻度を比較。
- いきなり全面導入せず診断・PoCに落とす。

## Counter Techniques by Bias

| Bias | Weak Counter | Strong Counter |
|---|---|---|
| 正常性 | 「危険です」 | 「前回と今回の条件差を見ましょう」 |
| 確証 | 「間違っています」 | 「反証になるケースを一つ確認しましょう」 |
| サンクコスト | 「諦めましょう」 | 「今から始めるとして同じ選択をしますか」 |
| 権威 | 「有名なだけ」 | 「その成功条件は自社にありますか」 |
| 損失回避 | 「怖がりすぎ」 | 「避けたい損失と、別の損失を並べましょう」 |
| 後知恵 | 「あなたも気づいていない」 | 「当時見えていた情報に戻しましょう」 |
| ハロー | 「でも性格が悪い」 | 「成果と副作用を別の評価軸で見ましょう」 |
| フレーミング | 「その言い方はずるい」 | 「目的と手段を分けて言い換えましょう」 |

## Result Screen Example

```text
今回検出された偏り:
- 前回と今回の警告を同一視していた
- ここまでの投入コストが判断に影響していた
- 顧客要求が「全機能必要」と過剰に解釈されていた
- QA側もリスクを一括りにして全面停止へ寄っていた

使えた技法:
- 前回との差分比較
- 顧客要求のスコープ分解
- ロールバック可否によるリスク分類
- 面子を守る退路提示
```
