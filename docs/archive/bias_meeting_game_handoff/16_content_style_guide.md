# Content Style Guide

## Tone

- 具体的、現場感、少し辛辣でもリアル。
- 抽象的な教訓より、Slack、PR、CI、会議の発言として見せる。
- 説教臭くしない。
- 登場人物を馬鹿にしない。
- バイアス名を会話中に連呼しない。

## Avoid

- 「ゲームの本質は〜」のような抽象語り。
- 「あなたは正常性バイアスです」のようなラベル貼り。
- 一方的に愚かな敵。
- 技術用語だけで中身がない会話。
- 正論だけで勝てる展開。
- 教材の穴埋め問題のような選択肢。

## Prefer

- 「前回と今回の失敗条件を比較したいです」
- 「設計全体ではなく、legacy条件だけ確認したいです」
- 「顧客が今月中に必要としている範囲を分けましょう」
- 「ロールバック不能な変更だけ先に切り出しましょう」
- 「議事録では、これはflakyとは記録しない方がいいです」

## Character Voice Guidelines

### PM

- 現実的、納期寄り、責任感あり。
- 評価や説明責任を恐れている。
- 少し言い訳っぽいが、悪人ではない。

Example:

```text
ここで止める判断をすると、顧客説明とリスケの影響がかなり大きいです。
軽微な懸念なら、予定通り進めたいです。
```

### Tech Lead

- 技術的自信がある。
- 断定が強い。
- 設計批判には敏感。

Example:

```text
このパターン自体は前職でも運用していました。
いま落ちているのは、実装というよりテストデータの揺れだと思います。
```

### QA

- 具体的な再現条件を持っている。
- 過去障害で傷ついている。
- 全面停止に寄りやすい。

Example:

```text
stagingで3回再現しています。
前回も「軽微」として進めて、本番で請求が止まりました。
```

### Sales

- 顧客信用と売上を守りたい。
- 技術詳細は浅いが、顧客の状況を知っている。
- 敵に見えるが味方化可能。

Example:

```text
延期そのものより、説明できない延期がまずいです。
何を出せて、何を来週にするのかが言えれば、交渉できます。
```

### CTO

- 時間を気にする。
- 事業速度とリスクの両方を見る。
- 技術リスクだけの議論には飽きる。

Example:

```text
リスクは分かりました。
では、顧客影響とロールバック可否を踏まえて、何を出して何を止めるのかを決めましょう。
```

## Document Writing Guidelines

### Slack

- 短文。
- 温度感が出る。
- 認識ズレや圧力が出る。

Example:

```text
PM 佐伯:
CI赤いですが、前回も似た感じだったのでGoで見ています。
今日中に最終確認だけお願いします。
```

### Jira

- 表面上は淡々としている。
- 優先度の誤分類やコメントにズレを入れる。

Example:

```text
Priority: P2
QA: stagingで3回再現。legacy_plan=trueで500。
Tech Lead: リリース後cleanupで対応可能。
```

### PR

- 技術的な懸念を短く。
- あとで効く「見落とされた指摘」を置く。

Example:

```text
Reviewer:
legacy customer の tax_category が null の場合、migration後に参照エラーになりませんか？
```

### CI Log

- 具体条件を明示。
- 読めばflakyではないと分かる。

Example:

```text
legacy_plan=true
tax_category=null
invoice_export_enabled=true
```

### Customer Email

- 営業の解釈と実際の要求にズレを作る。
- 顧客が悪者にならないようにする。

Example:

```text
今月中に必要なのは請求書出力です。
新決済フロー全体は来月初旬でも問題ありません。
```

## Choice Writing Guidelines

選択肢は、単純に正解/不正解にしない。

悪い：

```text
A. 正しい証拠を出す
B. 間違った証拠を出す
```

良い：

```text
A. CIログを強く突きつける
B. CIログで失敗条件を質問する
C. 設計全体ではなくlegacy条件に絞る
D. CPUスパイクログを出す
E. いったんピン留めする
```

## Failure Writing

失敗しても、プレイヤーを馬鹿にしない。

悪い：

```text
あなたは間違えた！
```

良い：

```text
黒瀬は少し眉をひそめた。
「そのCPUスパイクは別チームの負荷テストです。今回のmigrationとは関係ありません」
会議の空気が少し黒瀬側に傾く。
```

## Result Writing

リザルトでは、バイアス名より先に、何が起きたかを書く。

Example:

```text
見落とされた前提:
- CI失敗がflakyとして議事録に残った。
- 顧客が全機能を今月必要としている、という解釈が修正されなかった。

今回関係した偏り:
- 前回と今回を同一視した判断
- 過去の投入コストに引っ張られた判断
- 顧客要求の範囲を広く解釈した判断
```
