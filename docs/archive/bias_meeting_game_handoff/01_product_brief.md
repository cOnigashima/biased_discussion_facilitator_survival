# Product Brief

## Working Title

候補：

- Go / No-Go
- Release Judgment
- 認知デバッグ会議
- Bias Review
- 炎上前夜
- Meeting Debugger
- Premise Lock

## One-Liner

ソフトウェア開発プロジェクトの会議・レビュー・障害対応で、Slack、Jira、PR、ログ、顧客メールを読み解き、誤った前提が固まる前に会議を動かす、探索 + 会議バトル型ADV。

## Product Shape

| 項目 | 内容 |
|---|---|
| Genre | 推理ADV / 会話バトル / 開発現場シミュレーション |
| Core Loop | 探索 → 証拠化 → 会議バトル → 議事録修正 → 結果分岐 |
| Audience | ソフトウェアエンジニア、PM、EM、QA、SRE、ITコンサル、技術顧問 |
| Secondary | 逆転裁判・ダンガンロンパ系の会話推理が好きな人 |
| Tone | 現場感、具体性、会議の嫌なリアル、でもキャラを馬鹿にしない |

## Product Promise

プレイヤーは、混乱した開発現場の情報を読み、発言・証拠・議事録・利害のズレを見抜く。会議の中で、正しいカードを正しいタイミングと出し方で使い、強行リリース、過剰延期、責任押し付け、技術宗教戦争、顧客圧などを、より現実的な判断に変える。

## Core Fun Hypotheses

まだ未検証。MVPで検証する。

| 仮説 | 内容 | 検証する体験 |
|---|---|---|
| H1: 探索の面白さ | Slack/Jira/PR/CI/顧客メールを読んで違和感を見つけるのが楽しい | 「この発言、資料とズレてる」 |
| H2: 発言を突く快感 | 会議中の発言の怪しい箇所に証拠を当てるのが気持ちいい | 逆転裁判的な刺す感覚 |
| H3: タイミングの面白さ | すぐ反論するか、議事録にピン留めして後でコンボにするか迷う | 「今出すと荒れるかも」 |
| H4: 会議を動かす快感 | 出す/止めるの二択を、分割リリースなどに変える | 会議の流れが変わる |
| H5: 人間の扱い | 正論だけでなく、共感・退路・質問で人を動かす | 敵だと思った人が味方になる |
| H6: 開発現場あるある | flaky test、Feature Flag、rollback、PR懸念無視、Go/No-Goが刺さる | エンジニアが「ある」と感じる |

## MVP Goal

第1ステージのみで、以下を検証する。

- 探索資料を読む行為が退屈ではないか。
- 探索で得た証拠を会議バトルで使う快感があるか。
- 証拠選択だけでなく、出し方・タイミング・議事録修正が意味を持つか。
- 結果分岐に納得感があるか。
- ソフトウェア開発現場のリアリティがゲーム体験を強くしているか。

## MVP Scope

Included:

- テキストADV形式
- 探索資料 8〜10個
- 証拠カード 6〜8枚
- 罠カード 2枚程度
- 会議バトル 5ラウンド
- 議事録修正フェーズ
- 4〜5種類のエンディング
- リザルト画面
- 簡易フローチャート

Excluded initially:

- 大規模なキャラ育成
- 複数章の完全分岐
- 複雑な3D演出
- ランダム生成会議
- バイアス学習コンテンツの前面化
- マルチプレイ

## Non-Goals

- バイアス名を当てる教育クイズにしない。
- 「論破して勝つ」だけのゲームにしない。
- 開発現場の嫌な人を馬鹿にするだけのゲームにしない。
- ソフトウェア開発用語を表層的に並べるだけにしない。
- 抽象的な道徳劇にしない。

## Product Risks

| リスク | 内容 | 対策 |
|---|---|---|
| 資料を読むのが退屈 | Slack/Jira/PRが長いだけになる | 短く、矛盾がある文書にする。ハイライト/カード化を入れる |
| 証拠クイズになる | 正解カードを選ぶだけになる | 出し方、タイミング、議事録、Heatを入れる |
| 説教臭くなる | バイアス学習教材に見える | バイアス名はリザルトや図鑑に後退させる |
| キャラが薄い | 偏っただけの人になる | 実績・利害・恐れ・一理ある主張を持たせる |
| 分岐制作量が爆発 | Detroit的に作りすぎる | 固定ノード + 状態値で台詞差分 |
| 開発現場リアリティが弱い | エンジニアに刺さらない | 実際のPR/CI/ロールバック/顧客スコープを具体化 |
