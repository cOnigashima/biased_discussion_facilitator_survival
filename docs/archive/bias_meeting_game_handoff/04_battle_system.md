# Battle System

## Summary

バトルパートは、会議中の発言・議事録・前提を操作するテキストバトル。

```text
発言
→ 怪しい箇所を選ぶ
→ 証拠/過去発言/質問/フレーム変換/退路提示を選ぶ
→ 出し方を選ぶ
→ 会議・キャラ・議事録の状態が変わる
→ 次の発言へ
```

## Battle Units

### Utterance / 発言

現在の会議で誰かが言う主張。

例：

```text
PM「前回も似た警告は出ましたけど、問題ありませんでしたよね」
```

発言には複数のターゲットがある。

| 発言部分 | 攻め筋 | 対応バイアス |
|---|---|---|
| 前回も | 前回との差分確認 | 正常性 |
| 似た警告 | 同一視の検証 | 正常性・確証 |
| 問題ありませんでした | 過去結果の過信 | 正常性 |

### Evidence Card / 証拠カード

探索パートで得た資料。

例：

- CI失敗ログ
- 顧客メール
- PRコメント
- staging障害ログ
- ロールバック手順
- Feature Flag設計

### Meeting Minute / 議事録

発言が蓄積される盤面。放置された主張は議事録に残り、前提化する。

```text
議事録ドラフト：
- CI失敗は既知のflaky test
- 顧客には今月中リリースが必要
- QAは全面延期を希望
```

### Pinned Statement / ピン留め発言

今すぐ突かずに保存した発言。後で証拠や別発言と組み合わせて使う。

```text
PM発言「前回と同じ警告」
+ Tech Lead発言「flaky test」
+ CIログ「legacy_plan=trueで再現」
= 「前回と同じでも、flakyでもない」
```

### Premise / 前提

会議が判断根拠として扱い始めた主張。

```text
発言
→ 未検証論点
→ 暫定前提
→ 合意済み前提
→ 決定根拠
```

この状態を **Premise Lock** と呼ぶ。

## Core Parameters

### Global Meeting Parameters

| Parameter | Meaning | Notes |
|---|---|---|
| Heat | 会議の荒れ具合 | 高すぎると証拠が通りにくい |
| Clarity | 論点の明確さ | 高いと良い選択肢が解放される |
| Support | プレイヤー案への支持 | 最終案の通りやすさに影響 |
| Time | 残り発言機会 | 質問や保留で消費 |
| Momentum | 会議の流れ | Release / Stop / Split / Blame など |
| Premise Lock | 怪しい前提の固定度 | 高いほど後で崩しにくい |

### Character Parameters

| Parameter | Meaning |
|---|---|
| Confidence | 自分の主張への確信 |
| Defensiveness | 攻撃されたときの反発 |
| Face | 人前で撤回しにくい度合い |
| Anxiety | 隠れた不安 |
| Authority | 周囲への影響力 |
| TrustToPlayer | プレイヤーへの信頼 |
| Stake | 守りたい利害 |

UIでは数値より自然言語表示もあり。

```text
PM 佐伯：強気 / 少し焦り / 面子あり
Tech Lead 黒瀬：防衛的 / 権威あり / 技術論に強い
QA 三村：不安 / 証拠あり / 全面停止寄り
```

## Player Commands

| Command | Description | Strength | Risk |
|---|---|---|---|
| 証拠提示 | 資料を出して主張を崩す | 明確な矛盾に強い | 出し方次第でHeat上昇 |
| 質問 | 前提確認を促す | 防衛心を上げにくい | Timeを消費 |
| 共感 | 相手の目的・恐れを認める | Trust上昇・Heat低下 | 論点は進まない |
| 比較 | 前回/今回、A案/B案を比べる | 正常性・アンカリングに強い | 資料不足だと弱い |
| フレーム変換 | 二択や言葉の誘導を崩す | 第三案を出せる | 抽象的だと流される |
| 退路提示 | 相手の面子を守って修正する | 権威者に効く | 代替案が弱いと失敗 |
| ピン留め | 今は突かず、後で使う | コンボ可能 | 放置しすぎると前提化 |
| 議事録修正 | 誤った要約を直す | 最終判断に効く | Supportが低いと却下 |
| 味方に振る | 他キャラに説明してもらう | 自分が言うより角が立たない | 味方の信頼が低いと不発 |

## Timing Patterns

### 即時反論

```text
PM「前回も同じ警告でした」
Player「前回はログインtimeout、今回はlegacy migrationです」
```

強いがHeatが上がりやすい。チュートリアル向き。

### 質問で揺らす

```text
Player「前回と今回の失敗条件を比較してもいいですか？」
```

Clarityを上げるがTimeを使う。

### 保留してコンボ

```text
PM「前回と同じ」
Tech Lead「flaky」
Player「前回と同じでも、flakyでもありません。legacy条件で再現しています」
```

強い反転感。待ちすぎるとPremise Lockが進む。

### 議事録修正

```text
自動議事録「CI失敗は軽微なテスト不安定」
Player「軽微とは記録できません。legacy条件で再現しています」
```

最終局面の締めに使える。

## Battle Patterns

| Pattern | 発言例 | 弱点 | 良い攻め方 |
|---|---|---|---|
| 矛盾撃ち抜き型 | 「これはflaky testです」 | 失敗条件が毎回同じ | 「条件付き実バグでは？」 |
| 前提崩し型 | 「前回も大丈夫だった」 | 前回と今回が違う | 「差分を見ましょう」 |
| スコープ分割型 | 「出すか止めるか」 | 機能ごとにリスクが違う | 「migrationだけ止める」 |
| 利害変換型 | 「延期したら顧客信用を失う」 | 顧客が必要なのは全機能ではない | 「請求書出力だけで約束を守る」 |
| 面子保護型 | 「この設計で問題ない」 | 一部条件漏れ | 「legacy条件だけ確認したい」 |
| 権威シールド剥がし型 | 「有名企業でもこの方式」 | 自社制約と前提が違う | 「運用体制は同じですか？」 |
| ブリリアントジャーク解体型 | 「厳しくしないと品質が落ちる」 | 品質基準と人格攻撃が混ざる | 「重大度分類と基準文書化」 |
| 後知恵裁判型 | 「あの時点で分かっていた」 | 当時情報と今の情報が違う | 「当時確認できた情報に戻す」 |
| 数字シールド崩し型 | 「KPIは改善しています」 | 指標が偏っている | 「拾っていない失敗を見る」 |
| 時間圧突破型 | 「もう時間がない」 | 決めることと保留が混ざる | 「今決めるものを分ける」 |

## Counterattacks

| Counterattack | Example | Effect |
|---|---|---|
| 関連性攻撃 | 「それ、今回の話と関係あります？」 | Trust低下、相手権威上昇 |
| 権威攻撃 | 「前職でも実績があります」 | Authority発動 |
| 時間圧 | 「今やる必要あります？」 | Time減少 |
| 責任転嫁 | 「QAがもっと早く言うべきでした」 | Heat上昇、Blame Momentum |
| 二択化 | 「つまり延期したいんですね？」 | Frame固定 |
| 感情化 | 「私が無責任だと言いたいんですか？」 | Defensiveness上昇 |
| 論点ずらし | 「そもそもテスト環境が不安定です」 | Clarity低下 |
| 多数派圧 | 「みんなGoでいいと言っています」 | Support低下 |

## Win Conditions

| Ending Type | Meaning |
|---|---|
| Bad | 強行リリースして障害、または会議決裂 |
| Normal | 全面延期。安全だが顧客信頼・社内評価に傷 |
| Good | 危険部分だけ延期し、必要機能だけ出す |
| Great | Goodに加え、次回ルールや判断基準を制度化 |
